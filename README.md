# 💰 Expenses Bot - Telegram-бот для учёта финансов + Mini App

## 📋 Краткое описание

Telegram-бот для личного учёта расходов и доходов с уникальной функцией **контролируемых переводов** между пользователями + **современный Mini App веб-интерфейс**. Отправитель может видеть, на что получатель тратит переданные средства, обеспечивая полную финансовую прозрачность.

### Ключевые возможности:
- ✅ Учёт личных доходов и расходов
- ✅ Переводы средств между пользователями бота
- ✅ Прозрачность расходов получателя для отправителя
- ✅ **Telegram Mini App** - современный веб-интерфейс внутри Telegram
- ✅ **Графики и статистика** - визуализация финансов с помощью Recharts
- ✅ **Responsive дизайн** - адаптивный интерфейс на любых устройствах
- ✅ Категоризация расходов
- ✅ Детальная статистика и отчёты
- ✅ Экспорт данных в Excel/PDF
- ✅ Мультивалютность
- ✅ Напоминания и уведомления

## 🗄️ Структура базы данных

### Основные таблицы

#### 1. `users` - Пользователи
```sql
- id: BIGINT PRIMARY KEY (Telegram User ID)
- username: VARCHAR(255)
- first_name: VARCHAR(255)
- last_name: VARCHAR(255)
- language_code: VARCHAR(10) DEFAULT 'ru'
- default_currency: VARCHAR(3) DEFAULT 'RUB'
- created_at: TIMESTAMP
- updated_at: TIMESTAMP
- is_active: BOOLEAN DEFAULT TRUE
```

#### 2. `transactions` - Все финансовые операции
```sql
- id: UUID PRIMARY KEY
- user_id: BIGINT (FK -> users.id)
- type: ENUM('income', 'expense', 'transfer_out', 'transfer_in')
- amount: DECIMAL(15,2)
- currency: VARCHAR(3)
- category_id: INT (FK -> categories.id)
- description: TEXT
- transfer_id: UUID (FK -> transfers.id) [nullable]
- created_at: TIMESTAMP
- transaction_date: TIMESTAMP
```

#### 3. `transfers` - Переводы между пользователями
```sql
- id: UUID PRIMARY KEY
- sender_id: BIGINT (FK -> users.id)
- recipient_id: BIGINT (FK -> users.id)
- amount: DECIMAL(15,2)
- currency: VARCHAR(3)
- description: TEXT
- status: ENUM('pending', 'completed', 'cancelled')
- remaining_amount: DECIMAL(15,2) -- Остаток у получателя
- created_at: TIMESTAMP
- completed_at: TIMESTAMP
```

#### 4. `transfer_expenses` - Расходы из переводов
```sql
- id: UUID PRIMARY KEY
- transfer_id: UUID (FK -> transfers.id)
- transaction_id: UUID (FK -> transactions.id)
- amount: DECIMAL(15,2)
- description: TEXT
- category_id: INT (FK -> categories.id)
- created_at: TIMESTAMP
```

#### 5. `categories` - Категории расходов/доходов
```sql
- id: SERIAL PRIMARY KEY
- name: VARCHAR(100)
- type: ENUM('income', 'expense')
- icon: VARCHAR(10) [emoji]
- is_system: BOOLEAN DEFAULT FALSE
- user_id: BIGINT (FK -> users.id) [nullable] -- для кастомных категорий
```

#### 6. `balances` - Текущие балансы (материализованная view)
```sql
- user_id: BIGINT PRIMARY KEY (FK -> users.id)
- currency: VARCHAR(3)
- total_balance: DECIMAL(15,2)
- own_balance: DECIMAL(15,2) -- свои деньги
- received_balance: DECIMAL(15,2) -- полученные переводы
- updated_at: TIMESTAMP
```

#### 7. `notifications` - Настройки уведомлений
```sql
- id: SERIAL PRIMARY KEY
- user_id: BIGINT (FK -> users.id)
- type: ENUM('daily_reminder', 'transfer_received', 'transfer_spent')
- enabled: BOOLEAN DEFAULT TRUE
- time: TIME [для напоминаний]
- created_at: TIMESTAMP
```

#### 8. `reports` - История сгенерированных отчётов
```sql
- id: UUID PRIMARY KEY
- user_id: BIGINT (FK -> users.id)
- type: ENUM('daily', 'weekly', 'monthly', 'custom')
- format: ENUM('pdf', 'excel')
- file_path: VARCHAR(500)
- period_start: DATE
- period_end: DATE
- created_at: TIMESTAMP
```

### Индексы для оптимизации
```sql
-- Транзакции по пользователю и дате
CREATE INDEX idx_transactions_user_date ON transactions(user_id, transaction_date DESC);

-- Переводы по отправителю/получателю
CREATE INDEX idx_transfers_sender ON transfers(sender_id, created_at DESC);
CREATE INDEX idx_transfers_recipient ON transfers(recipient_id, created_at DESC);

-- Расходы по переводу
CREATE INDEX idx_transfer_expenses_transfer ON transfer_expenses(transfer_id, created_at DESC);
```

## ⚠️ Edge Cases и обработка ошибок

### 1. Недостаточный баланс
```
Пользователь пытается потратить больше, чем есть на балансе:
→ Бот: "❌ Недостаточно средств. Ваш баланс: 500 RUB, попытка расхода: 700 RUB"
```

### 2. Перевод самому себе
```
Пользователь пытается перевести деньги самому себе:
→ Бот: "❌ Нельзя перевести деньги самому себе. Выберите другого получателя."
```

### 3. Получатель не найден
```
Пользователь вводит @username, который не зарегистрирован в боте:
→ Бот: "❌ Пользователь @username не найден. Попросите его запустить бота: @expenses_bot"
   [Отправить приглашение]
```

### 4. Отмена перевода
```
Отправитель может отменить перевод только если получатель ещё не потратил деньги:
→ Если не потрачено: "✅ Перевод отменён. 3000 RUB возвращены на ваш счёт."
→ Если частично потрачено: "⚠️ @bob уже потратил 1200 RUB. Вернуть остаток (1800 RUB)?
   [Да, вернуть] [Отмена]"
→ Если всё потрачено: "❌ Невозможно отменить. Все средства уже израсходованы."
```

### 5. Некорректный ввод данных
```
Пользователь вводит текст вместо числа:
→ Бот: "❌ Некорректная сумма. Введите число (например: 500 или 1500.50)"

Отрицательная сумма:
→ Бот: "❌ Сумма должна быть положительным числом."

Слишком большая сумма (> 1,000,000):
→ Бот: "❌ Сумма слишком велика. Максимум: 1,000,000 RUB"
```

### 6. Одновременные операции
```
Пользователь начинает добавлять расход, затем перевод:
→ Бот автоматически отменяет предыдущее действие:
   "⚠️ Предыдущая операция отменена. Начинаю перевод."
```

### 7. Потеря соединения с БД
```
→ Бот: "⚠️ Технические неполадки. Попробуйте через минуту. Ваши данные сохранены."
→ Логирование в Sentry/CloudWatch
→ Retry механизм для критических операций
```

### 8. Некорректная категория
```
Пользователь выбирает категорию дохода для расхода:
→ Система автоматически фильтрует категории по типу операции
```

### 9. Мультивалютные переводы
```
Отправитель работает в USD, получатель в RUB:
→ Бот: "Ваш баланс в USD: $100. Конвертировать в RUB по курсу 75.50?
   3000 RUB = $39.74
   [Подтвердить] [Отмена]"
```

### 10. Дубликаты транзакций
```
Защита от случайного двойного нажатия:
→ Если транзакция с такими же параметрами создана < 60 сек назад:
   "⚠️ Похожая транзакция уже добавлена. Добавить ещё одну?
   [Да] [Нет, это ошибка]"
```

## 🔐 Безопасность и приватность

### 1. Данные пользователей
- Шифрование чувствительных данных (AES-256)
- Хеширование Telegram ID для анонимизации
- GDPR compliance (право на удаление данных)

### 2. Аутентификация
- Только через Telegram (OAuth)
- Проверка подлинности через Telegram API
- Rate limiting (макс. 10 запросов/мин на пользователя)

### 3. Права доступа
- Пользователь видит только свои данные
- Отправитель перевода видит только расходы из СВОЕГО перевода
- Получатель не видит другие переводы отправителя

### 4. Аудит
- Логирование всех финансовых операций
- Immutable logs (нельзя изменить историю)
- Backup каждые 6 часов

### 5. Удаление аккаунта
```
/delete_account → Подтверждение → Все данные удаляются безвозвратно
(кроме обезличенной статистики для аналитики)
```

- Uptime (целевой SLA: 99.9%)

### Мониторинг

## 🛠️ Развёртывание

### Локальная разработка
```bash
# Клонировать репозиторий
git clone https://github.com/yourusername/expenses-bot.git
cd expenses-bot

# Создать виртуальное окружение
python -m venv venv
source venv/bin/activate  # Windows: venv\Scripts\activate

# Установить зависимости
pip install -r requirements.txt

# Запустить БД через Docker
docker-compose up -d postgres redis

# Применить миграции
alembic upgrade head

# Запустить бота
python main.py
```

### Production (Docker)
```bash
# Сборка образа
docker build -t expenses-bot:latest .

# Запуск всего стека
docker-compose up -d

# Проверка логов
docker-compose logs -f bot
```

### Переменные окружения (.env)
```env
# Telegram
TELEGRAM_BOT_TOKEN=your_bot_token_here

# Database

REDIS_URL=redis://localhost:6379/0

# Security
SECRET_KEY=your-secret-key-here
ENCRYPTION_KEY=your-encryption-key

# Features
ENABLE_NOTIFICATIONS=true
ENABLE_REPORTS=true
MAX_TRANSFER_AMOUNT=1000000

# Integrations (optional)

```
